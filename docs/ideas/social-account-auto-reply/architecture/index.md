# 架构与数据流

> 传输层（WebSocket）已定；其余实现细节仍为备忘，随开发迭代再收紧。

## 传输层：WebSocket（已定）

- **扩展不对外提供 HTTP API**：不在浏览器里起监听端口；由扩展**主动连接**后端的 **WSS**，在**同一条长连接**上完成会话。  
- **全双工**：不是「只有后端 → 前端」。扩展必须能 **→ 后端** 发送：新消息等**观测**、心跳、鉴权、以及每步 tool 的**执行回执**。后端 **→ 扩展** 下发：智能体产生的**调用指令**（工具名、参数、顺序等）。  
- **指挥权在后端**：编排（多轮推理、何时下发哪一步）在**服务端**的 agent 循环里完成；扩展负责执行与回传结果，**不把业务流程写死在前端**。  
- **与智能体架构的关系**：标准闭环是「**环境观察 → 模型推理 → 工具调用 → 工具结果再喂给模型**」。扩展上报新状态 = **合法的 observation**，不是「前端抢大脑」。要避免的是前端**自行决定整条工作流**、后端只偶尔被拉一下生成一句话——那种才和「服务端智能体编排」冲突。  
- **服务端 agent 循环**：多轮「模型 ↔ 工具结果」发生在**后端进程内**，不是用 HTTP **轮询**替代 WebSocket；长连接负责**实时推送指令与上报结果**。  

## 当前倾向：前端「类 API」，后端编排下发

- **前端（扩展）**：把可执行能力做成**类似对外 API 的模块**——每个能力有稳定名称、入参 / 出参约定（本地函数 + 统一注册表）。  
- **后端**：跑 **Pydantic AI** 与记忆体；推理中产生的**调用逻辑**序列化后经 **WebSocket** 发给扩展。  
- **执行**：扩展按序调用本地能力，再通过 **WebSocket** 把结果回传，供下一轮推理。

用一句话：**扩展连上后端后，在一条长连接上与「服务端大脑」对话**——后端推送「要驱动页面干什么」；扩展**真正实现** DOM / 点击 / 填表等，并**回传观测与执行结果**。

## 和「后端声明 tool、前端执行」的关系

Pydantic AI 里的 **tool 语义上对应「前端能力」**；**执行在扩展进程**。「类 API」指**能力边界**，便于与下发指令一一对应；**不是**指扩展对外提供 REST。

## 仍待细化（与协议 / 待决联动）

- **一轮下发**：单步 vs 多步调用计划；失败重试与幂等 id。  
- **是否区分**业务连接与调试通道（若需要，可在协议里拆 topic 或第二条连接）。  
- **多 Tab / 多账号**：上下文如何绑定到 tab、profile 或站点会话。  
- **类 API** 是否未来演进为扩展侧本地 HTTP：若做，安全模型单独立项。
