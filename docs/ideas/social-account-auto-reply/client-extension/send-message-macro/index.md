# 发送消息与 UI 宏

## 语义：一个「类 API」

- 对上暴露 **向指定用户 / 会话发送一条文本**（参数如：检索条件或会话 id、正文），与后端 Pydantic AI 的 **单个 tool** 对齐。

## 典型实现：连环 UI

- **无官方接口**时的常见路径：打开私信页 → 搜索某人 → 点进会话 → 输入框填入 → 发送。路径拟人、易对照人工操作。

## 宏 vs 原子 tool vs Hook 请求

| 做法 | 优点 | 缺点 |
|------|------|------|
| **扩展内一条宏**（一个 tool，内部多步 UI） | 一次调用；顺序、重试、超时在插件内统一；省 token | 宏内要有恢复逻辑；依赖扩展内日志排障 |
| **多原子 tool** 全交给 agent | 单步失败可再推理；易单测每步 | WS 往返多、慢；中途易被改版打断 |
| **Hook 站点自有请求** | 常比纯点击稳、快 | 依赖站内实现，升级仍可能碎 |

**默认建议**：扩展内 **宏** 实现发送；回执带 **阶段标识**（如搜索无结果、输入框未找到），便于后端改策略。排障需要时再拆 **可选原子 tool**。
