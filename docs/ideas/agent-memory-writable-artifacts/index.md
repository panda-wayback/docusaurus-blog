# 智能体、记忆体与「可编辑的书写物」（需求想法）

> 仅**需求与方向**备忘，未涉及具体技术选型或实现。

## 动机

- 智能体要「能解决问题、显得聪明」，往往**离不开记忆体**；但**向量数据库**一类方案对使用者来说偏**抽象**，难直觉地理解「记忆里到底有什么、怎么改」。  
- 希望有一种**像博客或连续文档**的载体：既能承载记忆体工作时的产出，又让人类能**读、写、改**——而不是只在黑盒里调 embedding。

## 设想（大致感觉）

1. **智能体**仍配有**记忆体**（底层是否向量、图、其它存储可后议）。  
2. **记忆体不仅「存」，还要能「书写出来」**：把当前关心的内容、整理过的结论、与用户或任务相关的备忘，**持续写成可读文本**（类似博客、笔记流、项目 wiki）。  
3. **记忆体工作时**会**读取**这些已书写的内容作为依据之一（再配合检索、图关系等其它手段——「用图记忆或其它东西」都开放）。  
4. **人类编辑这些书写物 ≈ 直接改记忆体的语义**：改一段文字、删一篇、合并章节，就相当于调整智能体后续会依赖的「外在化记忆」。  

一句话：**记忆体外化为可编辑文本；文本是记忆的第一公民界面。**

## 目标效果（用起来是什么样）

- **日常**：你**持续指挥记忆体**、或**与智能体对话**，告诉它要怎么做、补充什么背景；智能体与记忆体在对话里协作。  
- **自动成文**：记忆体在过程中**持续沉淀**，把**对话里出现的内容**、以及对话里**挂上去的依赖**（链接、引用、约定），**自动整理成一篇篇可读文档**——这些文档就是记忆体后续所**依赖**的东西，而不是只躺在向量库里不可见。  
- **纠错很直观**：一旦行为不对、记忆有偏，你**不必**在黑盒里「调参数」；**直接改对应文档**（删、改、合并），就等于**改记忆体里的记忆**，下一轮智能体就会按新文档来。  

整体感受：**对话产生文档，文档即记忆；改文档即改记忆**，上手路径短、也符合「先写清楚再执行」的习惯。

## 开放问题（以后再想）

- 书写物的粒度（按会话 / 按用户 / 按项目）、版本与冲突。  
- 与纯向量检索、知识图谱的分工与同步。  
- 和本博客（Docusaurus）这类站点的关系：是同一类「文档」还是仅类比。
